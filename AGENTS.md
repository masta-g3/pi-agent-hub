# pi-agent-hub Agent Notes

- Keep this package Pi-native and small; do not import Agent Deck architecture unless explicitly requested.
- Groups are implicit session labels, not first-class records or selectable headers: `g` moves the selected session, `G` renames that session's current group label globally, and `K`/`J` reorder within the current group. Dashboard shortcuts are optimized for common actions: `r` opens restart choices (`r` selected, `n` new conversation, `a` all), `R` renames the selected session, and `N` syncs the hub title from Pi's `/name`. Custom dashboard shortcuts must stay config-driven Pi text sends, not shell commands or macro/event systems. Keep `e` as a hidden rename alias and `Alt+N` as a sync-name compatibility alias.
- Session row order is persisted/user-controlled via `src/core/session-order.ts`; do not reintroduce status/title sorting or a separate stopped-session section.
- TUI rendering must remain pure and testable. Any ANSI styling must be width-safe through the theme/layout helpers; when using left/right columns, reserve width for right-side status badges/counts instead of letting long titles or group names truncate them away. Use `src/tui/text-input.ts` for single-line text editing and `src/tui/form.ts`/`renderForm()` for editable dialogs instead of one-off dialog state.
- For Skills/MCP project state, write the final selection once. Do not launch parallel per-item JSON read-modify-write updates. The TUI pickers target the selected session's primary `cwd`; if no session is selected, they fall back to the dashboard cwd. In multi-repo sessions, Skills/MCP attach to the primary repo only, not additional repos or the synthetic workspace. Skill pool path editing lives in the `s` picker on `Alt+E`; `←`/`→` switch Enabled/Available columns and `Tab` remains an alias; keep plain printable keys, including `e`, available for picker search.
- Clipboard support is optional best-effort; attach/switch flows must still display the exact tmux command.
- Keep inside-tmux attach tmux-native: switch with `src/core/tmux.ts` helpers and do not stop/restart the TUI or add PTY attach unless outside-tmux return semantics are explicitly requested. If a tmux return path can recreate the dashboard, preserve `dashboardEnv()` so custom `PI_*` state dirs are not lost. `Alt+R` rename-from-session intentionally round-trips through the dashboard action handoff and should use the explicit dashboard rename dialog; dashboard-only `R`/hidden `e` use the footer prompt. Do not add a parallel in-session rename UI just to hide the brief dashboard flash.
- Tmux chrome must override both `*-style` and `*-format` options when avoiding inherited global colors; themes can embed ANSI/style directives in formats, not just styles. For themed footers, prefer Pi `statusLineBg` before `border` so Catppuccin border/accent colors do not become unreadable full-bar backgrounds.
- The extension can load via both `pi install` and managed-session `--extension`; keep registration idempotent and clear active guards on `session_shutdown`.
- `pi-hub` is the primary daily-use CLI command; keep `pi-agent-hub` as a compatibility command and keep package/runtime/tmux state names as `pi-agent-hub` unless a full package rename is explicitly requested. Keep package-global state under `PI_AGENT_HUB_DIR` or `<PI_CODING_AGENT_DIR>/pi-agent-hub`; do not use `<PI_CODING_AGENT_DIR>/sessions`, which is Pi's own conversation session directory. Multi-repo session workspaces live under `<PI_AGENT_HUB_DIR>/workspaces/<session-id>` and contain symlinks only; repo picker history lives in one bounded `<PI_AGENT_HUB_DIR>/repo-history.json` file with no filesystem scanning. Hub-owned worktrees live under `<PI_AGENT_HUB_DIR>/worktrees`; support is limited to explicit create/finish/discard flows, including multi-repo sessions that use one branch name across all repos, unless broader Git management is explicitly requested.
- Managed session bootstrap belongs in global `config.json` as `session.prelude` and applies only before managed `pi` starts; do not hardcode macOS keychain/SSH/direnv behavior or run it for the dashboard/direct TUI.
- For session deletion, use `src/app/delete-session.ts`, pause any active refresh loop first, and never delete Pi conversation/session files. Normal delete must not remove hub-owned worktree directories; use the explicit finish/discard flows in `src/app/worktree-session.ts`. For worktree finish/discard, preflight Git cleanliness before stopping parent/subagent tmux sessions, process additional repos before the primary repo, and keep workspace `.pi` pointed at the source repo so Pi state does not dirty hub-owned worktrees.
- Do not run `npm test` and `npm run package:check` concurrently: both rebuild `dist`, which can create false module-resolution failures during tests. If the user is actively using an installed/linked `pi-hub`, avoid build/package commands unless they explicitly approve; prefer `npx tsc -p tsconfig.json --noEmit` for non-disruptive validation.
- Optional `kind: "subagent"` registry rows are dashboard compatibility metadata owned by `pi-tmux-subagents`: keep them visually nested and short in the left pane (agent name only), keep task text in details/filtering, and do not enable normal session lifecycle/group/order actions on them.
- Optional `session-metadata/<session-id>.json` files are extension-owned transient display state: do not persist them into `registry.json`, do not use them for liveness/status counts/title sync, and clean them up with session deletion.
